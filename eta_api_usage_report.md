# تقرير استخدام واجهات برمجة التطبيقات الخاصة بمصلحة الضرائب المصرية (ETA API Usage Report)

يوضح هذا الملف جميع الـ APIs الخاصة بمنظومة الفاتورة الإلكترونية (ETA) التي تم دمجها، و**كيفية استخدامنا الفعلي لها** والأماكن التي يتم استدعاؤها منها داخل التطبيق (عبر `etaService.ts`). تمت مراجعة هذا التقرير ليتوافق مع أحدث إصدارات **Egyptian eInvoicing SDK**.

## أولاً: الـ APIs المستخدمة في المشروع وكيفية استخدامها ✅

### 1. المصادقة للأنظمة (Common & Authentication)
| الـ API (المسار) | كيفية وشكل الاستخدام في النظام |
|---|---|
| `POST /connect/token` | **الاستخدام:** يُستدعى عبر دالة `getToken()` قبل أي طلب آخر. <br> **طريقة العمل:** يقوم بإرسال `grant_type=client_credentials` و `client_id` و `client_secret`. وبمجرد استرجاع الـ Access Token، نقوم بـ **تخزينه مؤقتاً (Caching)** في الذاكرة لمدة 50 دقيقة (لتجنب الحظر لطلب التوكن مع كل مكالمة)، وتُرفق هذه التوكن كـ `Bearer` في هيدر أي طلب لاحق للـ API عبر نطاق الصلاحيات `InvoicingAPI`. |

### 2. أنواع الوثائق المُتاحة للإرسال (Document Types)
| الـ API (المسار) | كيفية وشكل الاستخدام في النظام |
|---|---|
| `GET /documenttypes` | **الاستخدام:** جلب جميع أنواع الوثائق الأساسية المتاحة كالفاتورة والإشعار وغيرها. |
| `GET /documenttypes/{typeId}` | **الاستخدام:** عرض الخصائص لنوع وثيقة محدد وتفاصيله (مثل إصدارات الفاتورة). |
| `GET /documenttypes/{typeId}/versions/...`| **الاستخدام:** الاستعلام عن نسخة محددة للوثيقة للتحقق من تحديثات بنيتها (Schema). |

### 3. إرسال الوثائق وتغيير حالاتها (eInvoicing Submissions)
*ملاحظة: المنظومة تدعم نوعيات متعددة للوثائق مثل (I للفوترة العادية، C للإشعار الدائن، D للإشعار المدين)، وكذلك فوترة التصدير (EI, EC, ED)، وتدعم صيغتي JSON و XML. نحن بالمشروع نعتمد على صيغة JSON السريعة.*

| الـ API (المسار) | كيفية وشكل الاستخدام في النظام |
|---|---|
| `POST /documentsubmissions` | **الاستخدام:** يُستدعى عبر `submitDocuments(documents)`. <br> **طريقة العمل:** هذا هو الـ Endpoint الأساسي والمحوري، حيث يتم إرسال مصفوفة المستندات الموقعة إلكترونياً (JSON Format) لجميع الأنواع: <br>- الفواتير العادية (**I**)<br>- الإشعارات الدائنة والمدينة (**C, D**)<br>- فواتير التصدير (**EI, EC, ED**) |
| `GET /documentsubmissions/{submissionId}` | **الاستخدام:** يُستدعى عبر `getSubmission()`. <br> **طريقة العمل:** بعد إرسال حزمة الفواتير، نستخدم معرف الدفعة `SubmissionId` للاستعلام عن الفواتير المقبولة `AcceptedDocuments` وتلك المرفوضة من الـ Validation. |
| `PUT /documents/state/{uuid}/state` | **الاستخدام:** يُستدعى لإجراءات الإلغاء/الرفض `cancelDocument` أو `rejectDocument`. <br> **طريقة العمل:** لتغيير حالة الفاتورة بتمرير `status: 'cancelled'` أو `rejected` مع الـ `reason`. |
| `PUT /.../decline/cancelation` | **الاستخدام:** يُستدعى عبر `declineCancellation`. <br> **طريقة العمل:** لحماية الشركة في حال محاولة الطرف الآخر الإلغاء نرسل رفضاً فنمنعه. |

### 4. تصفح وإدارة الفواتير الواردة والصادرة (Documents)
| الـ API (المسار) | كيفية وشكل الاستخدام في النظام |
|---|---|
| `GET /documents/search` | **الاستخدام:** يُستدعى عبر دالة `searchDocuments()`. <br> **طريقة العمل:** يُستخدم كبديل لـ Webhook حيث يتم البحث عن الفواتير الحديثة. يتم تمرير Parameter الـ `direction` وغيرها من التواريخ لجلب الفواتير بنظام التصفح. |
| `GET /documents/{uuid}/details` | **الاستخدام:** يُستدعى عبر `getDocumentDetails(uuid)`. <br> **طريقة العمل:** يُرجع التفاصيل الداخلية والبنود `invoiceLines` ونسبة الضرائب ونتائج الـ Validation. |
| `GET /documents/{uuid}/raw` | **الاستخدام:** يُستدعى عبر `getDocument(uuid)`. <br> **طريقة العمل:** لسحب ملف الـ JSON الموقّع رقمياً لفحصه برمجياً أو توثيقه. |
| `GET /documents/{uuid}/pdf` | **الاستخدام:** يُستدعى عبر `getDocumentPrintout(uuid)`. <br> **طريقة العمل:** لجلب وثيقة PDF الرسمية وإرسالها للمستخدم لعرضها وحفظها. |
| `GET /documents/recent` | **الاستخدام:** يُستدعى عبر `getRecentDocuments()`. <br> **طريقة العمل:** لسحب الوثائق الحديثة فقط لعرضها في الشاشة الرئيسية (Dashboard). |

### 5. نظام التكويد والأصناف (Item Codes Management)
| الـ API (المسار) | كيفية وشكل الاستخدام في النظام |
|---|---|
| `POST /codetypes/requests/codes` | **الاستخدام:** طلب ربط وإصدار كود EGS جديد برقم داخلي وGPC Code. |
| `GET /codetypes/requests/my` | **الاستخدام:** متابعة حالة أكواد الـ EGS المرفوعة للاعتماد (Approved, Rejected, Submitted). |
| `PUT /codetypes/requests/codeusages` | **الاستخدام:** استخدام كود EGS مسجل بشركة أخرى (Code Reuse / Linking). |
| `GET /codetypes/{codeType}/codes` | **الاستخدام:** فلترة وبحث مستمر عن أكواد الـ GS1 أو الـ EGS المتاحة لاستخدامها أثناء إدخال سطور الفاتورة. |
| `PUT /.../codes/{requestId}` | **الاستخدام:** تعديل طلب قيد المعالجة، أو `PUT` على كود معتمد وعامل بشكل نهائي لتحديث وصفه وغيره. |

### 6. الإشعارات والحزم (Notifications & Packages)
| الـ API (المسار) | كيفية وشكل الاستخدام في النظام |
|---|---|
| `GET /notifications/taxpayer` | **الاستخدام:** جلب الإشعارات كاستلام إخطارات الإلغاء والرفض للأطراف المشتركة عبر المنظومة. |
| `POST /documents/packages/requests` | **الاستخدام:** طلب ضغط (Export) للفواتير وحفظها كملفات ZIP في المنظومة. |
| `GET /documents/packages/{packageId}` | **الاستخدام:** الدالة التي تقوم فعلياً بتنزيل الحزمة للمستخدم (يُستخدم عادة في نهاية الشهر). |

## ثانياً: الـ APIs في المنظومة التي لم نقم باستخدامها ❌

| الخدمة | المسارات (Endpoints) غير المستخدمة | التفسير ولماذا لم تستخدم |
|---|---|---|
| **الإيصالات الإلكترونية B2C** | مجموعة `/receipts` و `/receiptsubmissions` | النظام الحالي مبني بشكل أساسي للفوترة الضريبية بين الشركات (B2B)، ونظام الـ e-Receipt يتطلب آلية مرتبطة بالـ POS. |
| **دعم الـ XML بشكل حصري** | `/documentsubmissions` بهيدر XML | برمجياً نعتمد على JSON خفيف الوزن للمنظومة بدلاً من دمج دوال تحويل الـ XML المكلفة (على الرغم من أننا قد قرأنا في SDK إمكانية ذلك). |
| **التفويضات (Delegations)** | `POST /delegations` وغيرها | إضافة مفوض أو محاسب جديد للشهادة تتم يدوياً من خلال البوابة لتلافي المشاكل الأمنية. |
| **تفاصيل الممول** | `GET /taxpayer/info/{rin}` | البحث عن الممولين الآخرين لمعرفة ما إذا كانوا مسجلين أو حالة التسجيل الخاصة بهم مُستثنى حالياً. |
| **حالة النظام** | `GET /system/status` | يمكن الاعتماد عليها مستقبلاً لإظهار حالة خوادم المصلحة الحية (Up/Down). |
